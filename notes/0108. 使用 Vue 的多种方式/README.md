## 2. 🫧 评价

这一节的核心价值不是教你某个 API，而是帮你建立“按场景选架构”的意识。Vue 不是只能做 SPA，也不是上来就该全套工程化。它被设计成可以从一段脚本渐进扩展到 SSR、SSG、跨端和自定义渲染器的框架。

## 3. 🤔 为什么 Vue 官方一直强调“没有一种方案适合所有场景”？

因为前端项目的目标差异太大了。

有的页面只是给已有后端页面加一点交互；有的应用需要长会话、复杂状态和丰富前端路由；有的站点更在意 SEO 和首屏速度；还有的需要把组件发给不同技术栈的宿主系统使用。

Vue 的设计目标不是强迫你进入某一种架构，而是让你在这些场景之间按需选择，在技术复杂度、开发体验和性能之间找到平衡。

## 4. 🤔 什么场景适合把 Vue 当成一个独立脚本来用？

### 4.1. 独立脚本

如果你的后端已经渲染好了大部分 HTML，前端只需要补少量交互，那么最简单的方式就是直接把 Vue 当一个独立脚本使用，不引入构建步骤。

这种场景下，Vue 更像“更声明式的 jQuery 替代品”。

适合：

- 后端模板页面增强
- 管理系统里的局部交互区
- 小部件或嵌入片段

不太适合：

- 大量组件复用
- 复杂路由
- 大规模状态管理

官方也提到过 `petite-vue` 这类更轻量的渐进增强方案，不过目前它已经不再积极维护。

### 4.2. 作为 Web Component 嵌入

如果你希望把一块 Vue 能力封装成标准自定义元素，嵌入旧系统、静态 HTML 或其他框架应用中，那么就可以把 Vue 组件编译成 Web Components。

这种方式的重点不是“拿它代替整个应用架构”，而是增强跨技术栈复用能力。

## 5. 🤔 什么时候该选择 SPA、SSR 或 SSG？

### 5.1. SPA

如果应用具备这些特征：

- 丰富交互
- 深会话
- 复杂状态逻辑
- 页面切换频繁

那么 SPA 往往是最自然的选择。

Vue 在 SPA 路线上的工具链支持非常完整，包括：

- 客户端路由
- 快速构建工具
- IDE 支持
- DevTools
- TypeScript
- 测试工具

### 5.2. SSR

如果你需要：

- 更好的首屏性能
- 更好的 SEO
- 服务端直接返回可见 HTML

那么可以考虑 SSR。

SSR 的核心思路是：服务端先把 Vue 应用渲染成 HTML，客户端再激活为可交互应用。Vue 生态里通常会优先考虑 Nuxt 这类上层方案。

### 5.3. SSG

如果页面内容可以在构建时就确定下来，那么很多场景其实更适合 SSG。

官方特别提到 SSG 有两种风格：

- 单页 SSG：首屏后激活成 SPA，后续导航快，但前期 JS 成本更高
- 多页 SSG：每次导航重新加载页面，但 JS 更少，甚至可接近零 JS

单页 SSG 更适合重交互、深会话场景；否则，多页 SSG 往往是更朴素也更高性价比的选择。

VitePress 和 Nuxt 都支持 SSG。

## 6. 🤔 Web Components 和 Web 之外的场景又意味着什么？

Vue 不只用于浏览器页面本身。

在 Web 之外，官方列出的常见方向包括：

- Electron / Wails：桌面应用
- Ionic Vue：移动端应用
- Quasar / Tauri：跨桌面与移动端
- TresJS：3D WebGL 体验
- 自定义渲染 API：例如终端界面

这一节真正想说明的是：Vue 的核心能力并不和“浏览器 DOM 页面”强绑定。只要存在可以被渲染和更新的目标，Vue 的组件模型和响应式系统就有机会被复用。
