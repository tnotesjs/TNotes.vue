## 2. 🫧 评价

这一节真正重要的不是记住内部函数名，而是看懂 Vue 的核心优势来自“编译器和运行时协同”。它并不是一个只有运行时 diff 的普通虚拟 DOM 框架，而是会在编译期把大量信息提前编码进渲染结果里，让运行时更新尽量走捷径。

## 3. 🤔 什么是虚拟 DOM，Vue 为什么要用它？

虚拟 DOM 是用普通 JavaScript 数据结构在内存里描述 UI 的方式。一个最简化的 vnode 可以长这样：

```js
const vnode = {
  type: 'div',
  props: {
    id: 'hello',
  },
  children: [],
}
```

渲染器会根据 vnode 树去创建真实 DOM，这叫挂载（mount）；当状态变化后，新旧 vnode 树会被比较并把差异应用到真实 DOM，这叫更新（patch / diff / reconciliation）。

虚拟 DOM 的主要价值是：

- 让开发者声明式描述 UI
- 让复杂 DOM 操作交给统一渲染器处理
- 让组件、插槽、动态结构更容易组合

## 4. 🤔 一个 Vue 组件从模板到真实 DOM 会经历什么？

官方把渲染管线概括成 3 步：

1. 编译
2. 挂载
3. 更新

### 4.1. 编译

模板会被编译为渲染函数，也就是返回 vnode 树的函数。这个步骤可以提前在构建期完成，也可以在运行时即时编译。

### 4.2. 挂载

运行时渲染器执行渲染函数，创建 vnode 树，再据此生成真实 DOM。这个过程本身会作为响应式副作用执行，因此会自动追踪渲染时访问到的依赖。

### 4.3. 更新

当依赖变化时，渲染副作用重新运行，生成新的 vnode 树，渲染器再把它和旧树比较，只把必要的变更同步到真实 DOM。

所以 Vue 组件本质上可以理解成：

- 响应式状态
- 渲染函数
- 一个由响应式系统驱动的 DOM 更新副作用

## 5. 🤔 为什么 Vue 默认推荐模板而不是手写渲染函数？

因为模板对大多数场景既更好写，也更容易被优化。

官方给出的理由主要有这些：

1. 模板更接近 HTML，更容易复用已有结构，也更友好于设计协作和可访问性。
2. 模板语法是确定的，编译器可以做静态分析和编译时优化。
3. 大多数应用场景里，模板已经足够灵活而且性能很好。

渲染函数更适合：

- 高度动态的结构
- 可复用渲染逻辑组件
- 模板写起来反而别扭的场景

所以推荐顺序通常是：模板优先，必要时再进入渲染函数。

## 6. 🤔 Vue 的“带编译时信息的虚拟 DOM”到底优化了什么？

这是 Vue 区别于“纯运行时虚拟 DOM”框架的关键。

### 6.1. 缓存静态内容

对完全静态的节点，Vue 在首次渲染后会缓存对应 vnode，后续更新时直接复用，跳过重新创建和 diff。

### 6.2. patch flags

编译器会提前知道某个节点到底有哪些动态部分，然后把这些信息编码进 vnode：

```js
createElementVNode(
  'div',
  {
    class: _normalizeClass({ active: _ctx.active }),
  },
  null,
  2 /* CLASS */,
)
```

这里的 `2` 就是更新类型标记。运行时只要看到这个 flag，就知道这次只需要关心 `class`，不必做全量检查。

### 6.3. 树结构打平

Vue 会把一个区块（block）内所有动态后代单独收集起来。这样更新时不需要遍历整棵树，只要遍历那些真正可能变化的节点。

官方把这理解为“树结构打平”：

- 静态部分跳过
- 动态部分集中追踪
- `v-if`、`v-for` 等结构性指令会创建新的区块边界

### 6.4. SSR 激活收益

patch flags 和 block tree 不只是优化客户端更新，也会提升 SSR hydration 的效率：

- 单个元素可依据 patch flag 走更快路径
- 激活时重点遍历区块和动态节点，而不是整棵树

这也是为什么 Vue 的模板编译能力会直接影响运行时性能。
